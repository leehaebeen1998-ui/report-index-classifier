# 보고서 인덱스 분류 모듈 설계

## 목적

광고 보고서 업로드 시 보고서마다 컬럼 구조와 명칭이 달라도 자동으로 카테고리를 분류할 수 있는 인덱스 구조를 만든다.

완전한 단일 공통 인덱스로 모든 보고서를 강제하지 않고, 공통 로직과 브랜드별 예외 규칙, AI 보정 구조를 조합한다.

이 문서의 구현 대상은 완성형 업로드 응용 프로그램이 아니라 독립 모듈이다. 파일 업로드, 결과 확인 화면, 최종 보고서 다운로드, 스케줄러 연결은 2차 응용 프로그램의 책임으로 둔다.

## 모듈 경계

1차 인덱스 분류 모듈의 책임:

- 보고서 컬럼 자동 인식
- 강제 URL 매핑
- 캠페인 유형 분류
- 그룹명 분류
- 키워드명 분류
- 캠페인명 분류
- 일반 URL 분류
- 규칙 실패 시 AI 분석
- 사용자가 수정한 결과를 인덱스에 저장

2차 응용 프로그램의 책임:

- 보고서 파일 업로드
- 1차 인덱스 모듈 호출
- 분류 결과 확인
- 사용자가 수정
- 수정값 저장
- 최종 보고서 다운로드
- 추후 스케줄러/자동화 연결

## 핵심 원칙

계정 ID와 브랜드 ID는 분류 기준에서 제외한다.

계정 ID 또는 브랜드 ID는 다음 용도로만 사용한다.

- 업로드된 보고서의 인덱스 검색 범위 제한
- 브랜드별 예외 규칙 로드
- 사용자 수정값을 저장할 인덱스 범위 결정

처리 예시는 다음과 같다.

```text
보고서 업로드
-> 브랜드/계정 식별
-> 해당 브랜드 인덱스 로드
-> 0~6순위 분류 규칙 적용
```

브랜드/계정이 식별되어도 그것만으로 카테고리를 확정하지 않는다.

## 최종 분류 우선순위

```text
0순위 : 강제 URL 매핑
1순위 : 캠페인 유형
2순위 : 그룹명
3순위 : 키워드명
4순위 : 캠페인명
5순위 : 일반 URL
6순위 : AI 분석
```

## 전체 데이터 구조

인덱스는 공통 구조를 유지하되, 실제 데이터는 브랜드 또는 계정 범위로 나누어 저장한다.

```json
{
  "index_version": "1.0.0",
  "scope": {
    "brand_id": "brand_001",
    "account_ids": ["account_001"]
  },
  "column_aliases": {},
  "forced_url_mapping": [],
  "campaign_type_rules": [],
  "group_name_rules": [],
  "keyword_rules": [],
  "campaign_name_rules": [],
  "url_rules": [],
  "user_corrections": [],
  "ai_fallback": {
    "enabled": true
  }
}
```

## 컬럼명 자동 인식

보고서마다 컬럼명이 다르므로, 업로드 단계에서 원본 컬럼을 표준 필드로 매핑한다.

표준 필드는 다음을 기본값으로 둔다.

| 표준 필드 | 의미 |
| --- | --- |
| `account_id` | 계정 ID |
| `brand_id` | 브랜드 ID |
| `campaign_type` | 캠페인 유형 또는 매체 유형 |
| `campaign_name` | 캠페인명 |
| `group_name` | 광고 그룹명 또는 세트명 |
| `keyword_name` | 키워드명 |
| `url` | 랜딩 URL 또는 최종 URL |
| `creative_name` | 소재명 |
| `ad_text` | 광고 문구 |

컬럼 별칭은 인덱스에 누적할 수 있다.

```json
{
  "column_aliases": {
    "campaign_type": ["캠페인 유형", "매체", "광고상품", "channel"],
    "campaign_name": ["캠페인", "캠페인명", "campaign"],
    "group_name": ["그룹", "광고그룹", "광고 그룹명", "ad_group", "adset"],
    "keyword_name": ["키워드", "키워드명", "keyword"],
    "url": ["URL", "랜딩URL", "최종 URL", "landing_url", "final_url"]
  }
}
```

## 분류 규칙

### 0순위: 강제 URL 매핑

특정 URL이 반드시 특정 카테고리로 분류되어야 하는 경우 사용한다.

강제 URL은 모든 조건보다 우선한다.

```text
URL이 forced_url_mapping에 존재
-> 즉시 해당 category 반환
-> 캠페인 유형, 그룹명, 키워드명, 캠페인명 분석 생략
```

권장 매칭 방식은 정규화된 URL의 정확 일치다.

- scheme, host, path 기준 비교
- trailing slash 통일
- query string은 기본적으로 제거
- 필요 시 `include_query: true`로 예외 허용

### 1순위: 캠페인 유형

캠페인 유형 또는 매체 유형을 상위 조건으로 사용한다.

예시는 다음과 같다.

- 검색광고
- P-MAX
- 디맨드젠
- GFA
- Meta
- Naver SA
- Kakao
- ADN

캠페인 유형은 단독으로 확정하기보다, 이후 그룹명/키워드명/캠페인명 규칙의 적용 범위를 좁히는 용도로 쓰는 것을 권장한다.

예를 들어 `Naver SA`에서는 키워드 규칙을 강하게 적용하고, `Meta`나 `GFA`에서는 키워드 단계를 건너뛴다.

### 2순위: 그룹명

그룹명은 실제 운영 카테고리가 많이 반영되는 필드이므로 핵심 분류 기준으로 사용한다.

예시는 다음과 같다.

- 형사
- 마약
- 음주
- 성범죄
- 이혼
- 부동산
- 조세

권장 규칙 형태:

```json
{
  "field": "group_name",
  "match_type": "contains",
  "patterns": ["형사", "형사전문"],
  "category": "형사",
  "confidence": 0.9
}
```

### 3순위: 키워드명

검색 광고에서는 키워드명이 검색 의도를 잘 반영하므로 강한 분류 기준으로 사용한다.

단, 키워드 필드가 없거나 캠페인 유형상 키워드가 없는 매체에서는 이 단계를 건너뛴다.

예:

```text
Meta, GFA, P-MAX, 디맨드젠
-> keyword_name 없음
-> 키워드 규칙 생략
```

### 4순위: 캠페인명

캠페인명은 브랜드별 작명 규칙 차이가 크므로 그룹명보다 후순위로 둔다.

그룹명과 키워드명이 비어 있거나 불명확할 때 보조 기준으로 사용한다.

### 5순위: 일반 URL

일반 URL은 보조 기준이다.

강제 URL과 달리 절대 기준으로 처리하지 않는다.

예시는 다음과 같다.

- `/criminal/` 포함 -> 형사 가능성
- `/drug/` 포함 -> 마약 가능성
- `/divorce/` 포함 -> 이혼 가능성

일반 URL 규칙은 낮은 기본 신뢰도를 부여한다.

### 6순위: AI 분석

규칙 기반 분류가 실패했거나 신뢰도가 낮을 때만 AI 분석을 사용한다.

AI 분석 대상은 다음과 같다.

- 랜딩페이지 텍스트
- 소재명
- 광고 문구
- 페이지 title
- meta description

AI 분석 결과는 바로 영구 규칙으로 반영하지 않고, 사용자 확인 또는 수정 후 `user_corrections`에 저장한다.

## 신뢰도 정책

각 규칙은 `confidence`를 반환한다.

권장 기준:

| 범위 | 의미 | 처리 |
| --- | --- | --- |
| `0.90` 이상 | 매우 높음 | 자동 확정 |
| `0.70` 이상 | 보통 | 결과 표시, 필요 시 검토 |
| `0.70` 미만 | 낮음 | 다음 우선순위 또는 AI 분석 |

강제 URL 매핑은 항상 `1.0`으로 처리한다.

## 사용자 수정값 누적

사용자가 분류 결과를 수정하면 다음 정보를 저장한다.

```json
{
  "source": "user_correction",
  "category": "형사",
  "evidence": {
    "campaign_type": "Naver SA",
    "group_name": "형사_구속",
    "keyword_name": "형사변호사",
    "campaign_name": "검색_형사",
    "url": "https://example.com/criminal"
  },
  "created_at": "2026-06-25T00:00:00+09:00"
}
```

재사용 방식:

- 동일 URL이 반복 수정되면 강제 URL 후보로 제안
- 동일 그룹명 패턴이 반복되면 그룹명 규칙 후보로 제안
- 동일 키워드 패턴이 반복되면 키워드 규칙 후보로 제안
- 사용자 승인 후 정식 규칙으로 승격

## 추천 처리 플로우

```text
1. 보고서 업로드
2. 컬럼명 자동 인식
3. 브랜드/계정 식별
4. 해당 범위의 인덱스 로드
5. URL이 forced_url_mapping에 있는지 확인
6. 있으면 즉시 카테고리 확정
7. 없으면 캠페인 유형 확인
8. 캠페인 유형에 맞는 하위 규칙 필터링
9. 그룹명 확인
10. 키워드명 확인
11. 캠페인명 확인
12. 일반 URL 확인
13. 그래도 불명확하면 AI 분석
14. 결과, 신뢰도, 근거 반환
15. 사용자가 수정한 값은 인덱스에 누적 저장
```

## 분류 결과 포맷

분류 결과는 카테고리만 반환하지 않고, 판단 근거와 다음 액션을 함께 반환한다.

```json
{
  "category": "형사",
  "confidence": 0.9,
  "matched_priority": 2,
  "matched_rule_id": "group-criminal-001",
  "evidence": {
    "field": "group_name",
    "value": "형사_구속",
    "pattern": "형사"
  },
  "needs_review": false
}
```

## 운영 구조

AI가 초안을 생성하고 사용자가 예외만 수정하는 구조를 기본 운영 방식으로 둔다.

권장 운영 순서:

```text
초기 업로드
-> AI가 컬럼 별칭과 규칙 후보 생성
-> 규칙 기반 분류 실행
-> 불명확한 행만 사용자 검토
-> 사용자 수정값 누적
-> 반복 패턴을 정식 규칙 후보로 제안
```

이 방식은 초기에 모든 예외를 직접 설계하지 않아도 되고, 실제 업로드 데이터를 통해 인덱스 품질을 점진적으로 높일 수 있다.
